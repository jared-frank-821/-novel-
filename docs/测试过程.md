# My Novel 测试体系建设过程

## 一、背景与目标

在接入后端数据库（Supabase）之前，先建立完整的测试体系。目标是：
1. 为后续重构数据层提供安全保障
2. 覆盖核心业务逻辑（Zustand Store）
3. 覆盖 UI 组件交互（分类标签选择器）
4. 建立 CI 自动化流水线（GitHub Actions）

---

## 二、技术选型

### 为什么选 Vitest 而不是 Jest？

| 对比项 | Vitest | Jest |
|--------|--------|------|
| 启动速度 | Vite 原生，毫秒级 | 需要额外配置 Babel，较慢 |
| TypeScript 支持 | 开箱即用，无需额外配置 | 需要 babel-jest + ts-jest |
| Vite 生态集成 | 和 Next.js/Vite 项目无缝衔接 | 需要额外适配 |
| API 兼容性 | 100% 兼容 Jest API | — |

最终选择：**Vitest + React Testing Library + jsdom**

## 三、文件清单与说明

### 3.1 `vitest.config.ts`

测试运行器核心配置，放在项目根目录：

```typescript
export default defineConfig({
  plugins: [react({ jsxImportSource: 'react' })],
  test: {
    environment: 'jsdom',              // 模拟浏览器 DOM 环境
    setupFiles: ['./src/test/setup.ts'], // 测试前加载全局 mock
    globals: true,                     // 全局暴露 describe/it/expect/vi
    include: ['src/**/*.{test,spec}.{ts,tsx}'], // 只跑这些文件
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json', 'html'],
      include: ['src/lib/**', 'src/store/**', 'src/components/ui/select.tsx'],
    },
  },
  resolve: {
    alias: { '@': path.resolve(__dirname, './src') }, // 路径别名解析
  },
});
```

关键点解析：
- `environment: 'jsdom'`：Node.js 环境本身没有 DOM，jsdom 会在内存中模拟一个完整的浏览器环境，包括 `document`、`window`、`Element` 等，使得 React 组件可以在 Node.js 中渲染和测试。
- `setupFiles`：在每个测试文件运行之前先加载这个文件，用于放置全局 mock，确保浏览器 API 在 Node.js 中可用。
- `coverage`：配置 v8 代码覆盖率引擎，支持生成 text、json、html 三种格式的报告。

### 3.2 `src/test/setup.ts`

全局 Mock 文件，解决 Node.js 缺少浏览器 API 的问题：

```typescript
// Mock localStorage
const localStorageMock = (() => {
  let store: Record<string, string> = {};
  return {
    getItem: vi.fn((key) => store[key] ?? null),
    setItem: vi.fn((key, value) => { store[key] = value; }),
    removeItem: vi.fn((key) => { delete store[key]; }),
    clear: vi.fn(() => { store = {}; }),
    get length() { return Object.keys(store).length; },
    key: vi.fn((i) => Object.keys(store)[i] ?? null),
  };
})();
Object.defineProperty(globalThis, 'localStorage', { value: localStorageMock });
```

这里用的是 `Object.defineProperty` 而不是直接赋值 `global.localStorage = ...`，因为 `localStorage` 和 `matchMedia` 在全局对象上已经有属性描述符（`writable: false`），直接赋值会被静默忽略。`Object.defineProperty` 可以强制覆盖已有描述符。

Mock 了三套 API：
- **localStorage**：Zustand persist 中间件依赖此 API 持久化状态
- **matchMedia**：一些 UI 组件内部判断暗色模式时用到
- **ResizeObserver**：shadcn/ui 组件内部监听尺寸变化时用到

另外，抑制了 `console.error` 中的一些 React 内部警告（如 "not wrapped in act"），避免干扰测试输出。

### 3.6 `.github/workflows/ci.yml`

GitHub Actions 持续集成配置，放在 `.github/workflows/` 目录下：

```yaml
on:
  push:
    branches: [main, master]
  pull_request:
    branches: [main, master]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true  # 新流程开始时自动取消旧流程
```

流水线步骤：

```
Checkout → npm ci → ESLint → Type Check → Vitest → 上传覆盖率报告
```

`concurrency` 配置确保：当同一个分支同时有多个 push 时，只有最新的那个会跑完 CI，之前的会被自动取消，节省 GitHub Actions 分钟数。

## 四、遇到的坑与修复

### 坑 1：测试前状态没清干净

**现象**：`addNovel` 测试中，`novels` 数组是空的，断言失败。

**根因**：Zustand 的 `persist` 中间件将状态写入 `localStorage`，而 Zustand 内部合流（rehydration）是**异步**的。测试跑得太快，在异步合流完成之前就开始了，导致读到的状态不完整。

**修复**：`beforeEach` 里 `clearAll()` 后，`await new Promise(r => setTimeout(r, 0))`，等待一个 event loop tick，确保 persist middleware 完成初始化：

```typescript
beforeEach(async () => {
  useNovelListStore.getState().clearAll();
  await new Promise(r => setTimeout(r, 0));
});
```

### 坑 2：Zustand getState() 快照陷阱

**现象**：先 `const store = getState()` 获取快照，再 `store.addNovel()`，然后用 `store.novels` 做断言，断言失败。

**根因**：`store` 是 Zustand state 的快照（一个普通 JS 对象），不是响应式引用。`addNovel()` 调用后，store 内部的状态确实变了，但 `store.novels` 读的还是快照里的旧值。

**修复**：操作后重新调用 `getState()` 获取最新状态：

```typescript
// ❌ 错误写法
const store = useNovelListStore.getState();
store.addNovel('test');
expect(store.novels.length).toBe(1);  // 失败！

// ✅ 正确写法
useNovelListStore.getState().addNovel('test');
expect(useNovelListStore.getState().novels.length).toBe(1);  // 通过！
```

### 坑 3：updateTime 同一毫秒内相同

**现象**：断言 `expect(newUpdateTime).not.toBe(originalUpdateTime)` 失败，两个时间戳相同。

**根因**：`new Date().toISOString()` 的精度是毫秒级。在同一个 JS tick 内连续调用两次 `new Date()`，`Date.now()` 的返回值完全相同。

**修复方案一**：在两次操作之间等待 1ms，强制跨毫秒：
```typescript
await new Promise(r => setTimeout(r, 1));
useNovelListStore.getState().updateChapterText(1, '新内容');
```

**修复方案二**：简化断言，只验证数据正确，不依赖时间戳比较（测试中采用）：
```typescript
// 不比较 updateTime，只验证数据本身正确
expect(chapter.text).toBe('这是正文内容');
expect(chapter.wordCount).toBe(6);
expect(chapter.createTime).toBeTruthy();
expect(chapter.updateTime).toBeTruthy();
```

### 坑 4：重复文本选择器

**现象**：`screen.getByText('题材')` 失败，Found multiple elements。

**根因**：下拉菜单里有两个"题材"文本——一个是顶部的**分类筛选按钮**，另一个是**标签分组标题**（"题材"下的标签列表标题）。

**修复**：用 `getByRole` 精确匹配按钮角色：
```typescript
// ❌ 错误
const themeBtn = screen.getByText('题材');

// ✅ 正确
const themeBtn = screen.getByRole('button', { name: '题材' });
```
